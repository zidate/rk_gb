# GAT1400

## 目的
将 `GA/T 1400.4-2017` 沉淀为本仓库 GAT1400 / VIID 开发、联调、代码审核和协议符合性检查的基线文档。

## 模块概述
- **职责:** 负责 GAT1400 注册、注销、保活、校时、本地订阅 HTTP 服务、结构化对象上报、订阅观察者通知
- **状态:** ✅已有实现，后续开发和审核必须回到标准基线检查
- **规范基线:** `GA/T 1400.4-2017`、`GA/T 1400.3-2017`、`GA/T 1399`、`RFC 2045`、`RFC 2616`、`RFC 2617`、`RFC 2818`
- **最后更新:** 2026-03-28

## 规范基线

### 标准适用范围
- `GA/T 1400.4-2017` 适用于公安视频图像信息应用系统的规划设计、软件开发和接口协议符合性测试。
- 本部分定义的是接口分类、协议结构、资源 URI、消息配对、交互流程和安全性。
- 具体对象字段和 Schema 不在本部分展开，统一回到 `GA/T 1400.3-2017`。
- 分析系统对象和能力模型统一回到 `GA/T 1399`。

### 接口分类

| 类别 | 交互双方 | 典型内容 |
|------|----------|----------|
| 采集接口 | 采集设备/采集系统/分析系统 -> 视图库 | 自动采集视频片段、图像、文件、人车物场景上传 |
| 数据服务接口 | 应用平台/其他系统/分析系统 <-> 视图库 | 特征查询、人工采集、布控告警、订阅通知 |
| 级联接口 | 上下级视图库之间 | 查询、布控、订阅、通知 |
| 分析接口 | 应用平台 <-> 分析系统 | 任务下发、任务状态、分析结果、分析能力 |

### 协议共性约束
- 所有接口交互信息都应抽象为 REST 资源，并由 URI 唯一标识。
- 资源按树状层级组织，当前标准定义两大顶级资源族：`/VIID/*` 和 `/VIAS/*`。
- 交互连接方式应支持 HTTP 长连接和短连接；`Protocol` 应支持 `HTTP`，宜支持 `HTTPS`。
- HTTP 请求头应增加 `User-Identify`，携带请求者系统用户 ID 等身份属性。
- 接口消息 `Content-Type` 应设为 `application/*+JSON`。
- `GET` 请求在 `2XX` 成功场景下返回资源对象；非 `2XX` 场景返回 `ResponseStatus`。
- `POST/GET/PUT/DELETE` 的语义分别对应 Create / Retrieve / Update / Delete。

### 公共时序要求
- 注册失败后，应在 `300s` 内随机延迟后重新注册。
- 注册成功后，`90s` 内如果没有业务交互，应主动发送保活。
- 上传和通知在一定延迟时间内未收到响应时，应判定失败并重传。
- 连续多次重传失败后，应视为链路问题，缓存未发送数据，重新注册成功后补传。

### 消息交互安全性
- 双方应执行身份认证，认证过程遵循 `RFC 2617` 摘要认证。
- 宜遵循 `RFC 2818`，即支持 HTTPS/TLS。
- 摘要认证的 `nonce` 超过 `3600s` 后应重置。

### 现状提示
- 当前仓库的 GAT1400 重点是“下级 / 设备侧 lower client + 本地订阅接收”，不是完整的双向 VIID 资源服务器。
- 当前请求目标已经支持 `scheme + server_ip + server_port + base_path + request_timeout_ms` 统一建模，但实际传输仍只实现 `HTTP`，若平台要求 `HTTPS`，仍需补齐 TLS 链路。
- 标准要求的“失败缓存、重新注册、补传”已补入最小闭环：失败请求持久化到 `gat_upload.queue_dir`，注册成功和保活恢复后回放；但尚未形成独立 mock 联调测试。
- 抓拍桥接层当前是进程内内存队列，不落 flash；若编码/算法侧在 1400 服务启动前提交抓拍事件，会先留在内存队列，待 `GAT1400ClientService` 启动后再 drain 上传。
- `POST /VIID/APEs` 不是当前标准基线里的常规 lower client 上报接口，现已收口为显式兼容开关 `gat_upload.enable_apes_post_compat`，默认关闭。
- `/VIAS/*` 在本阶段明确不支持，避免被透传命令误送到失败补传队列里反复重放。
- 当前业务侧接入建议优先走 `LowerGAT1400SDK` 导出层，而不是自行再管理一套 `GAT1400ClientService` 实例；导出层现直接复用 `ProtocolManager::Instance()` 对应的运行时服务。
- 人脸抓拍、机动车抓拍这类设备能力当前不再建议直接拼 `LOWER_1400_POST_*` 调 1400 服务；建议由编码侧 / 算法侧通过 `GAT1400CaptureControl` 投递事件，再由 1400 模块统一映射和上传。

## 抓拍桥接使用说明

### 适用场景
- 编码侧 / 算法侧已经拿到一条“人脸抓拍”或“机动车抓拍”结果，并且手里还有关联图片、视频片段或其他文件。
- 业务侧希望只把抓拍结果交给 1400 模块，不自己关心 `PostFaces/PostImages/PostVideoSlices/PostFiles` 的调用顺序。

### 调用方职责
- 调用方负责产出真实抓拍结果，也就是 `GAT_1400_Face` 或 `GAT_1400_Motor`，以及可选的 `GAT_1400_ImageSet/GAT_1400_VideoSliceSet/GAT_1400_FileSet`。
- 调用方负责把二进制内容预先放进 `ImageSet.Data`、`VideoSliceSet.Data`、`FileSet.Data`；1400 模块不会再帮你读文件。
- 调用方最好补齐结构化对象里的 `SourceID`、`DeviceID`。如果没填，1400 服务只会尝试从关联图片 / 视频元数据里补最小来源关系。

### 推荐调用时机
- 抓拍算法已经出最终结果时立即调用。
- 不需要等 1400 模块单独 `init/start` 回调。当前桥接层会先在内存里排队，等 `GAT1400ClientService` 启动后自动消费。
- 如果设备进程可能重启，不要把“未提交到桥接层的抓拍结果”只放在临时内存里；桥接层本身也只保证进程内排队，不做 flash 持久化。

### 推荐调用方式
```cpp
#include "Media/GAT1400CaptureControl.h"

void ReportFaceCapture(const GAT_1400_Face& face,
                       const std::string& imageBase64,
                       const std::string& imageId,
                       const std::string& deviceId)
{
    GAT_1400_ImageSet imageSet;
    strncpy(imageSet.ImageInfo.ImageID, imageId.c_str(), sizeof(imageSet.ImageInfo.ImageID) - 1);
    strncpy(imageSet.ImageInfo.DeviceID, deviceId.c_str(), sizeof(imageSet.ImageInfo.DeviceID) - 1);
    strncpy(imageSet.ImageInfo.FileFormat, "Jpeg", sizeof(imageSet.ImageInfo.FileFormat) - 1);
    imageSet.Data = imageBase64;

    std::list<GAT_1400_ImageSet> imageList;
    imageList.push_back(imageSet);

    media::GAT1400CaptureControl::Instance().SubmitFaceCapture(face, imageList, std::list<GAT_1400_VideoSliceSet>(), std::list<GAT_1400_FileSet>(), imageId);
}
```

### 1400 侧当前处理顺序
1. 如果是人脸抓拍，先上报 `Faces`。
2. 如果是机动车抓拍，先上报 `MotorVehicles`。
3. 再依次上报 `Images`、`VideoSlices`、`Files`。
4. 每个 `ImageSet/VideoSliceSet/FileSet` 如果没带对应结构化对象，1400 服务会把本次事件的 `Face/Motor` 关联进去。

### 当前不做的事情
- 不负责从磁盘路径读取图片、视频、文件。
- 不负责抓拍算法本身，也不负责生成 `FaceID/MotorVehicleID/ImageID/VideoID/FileID`。
- 不负责断电恢复；桥接层只保存进程内内存队列，真正进入 1400 上传主链后，才会继续复用现有失败补传文件队列。

## 开发清单
- 确认资源 URI、HTTP 方法和查询字符串与 `GA/T 1400.4-2017` 一致。
- 确认对象结构、字段约束和列表包装来自 `GA/T 1400.3-2017` 或 `GA/T 1399`，不要在 1400.4 中臆造字段。
- 所有 JSON 请求应带 `Content-Type: application/*+JSON`。
- 所有对上请求都应带 `User-Identify`。
- 对需要鉴权的链路，必须走摘要认证；若平台要求 HTTPS，需要显式支持。
- `/<ID>/Data` 资源的消息体是 `Base64` 编码二进制流，不能直接裸传文件字节。
- 注册失败、保活超时、通知失败、上传失败必须检查是否具备重试、缓存、重注册和补传闭环。
- `ResponseStatus` / `ResponseStatusList` 的 `Id`、`StatusCode`、`StatusString`、`LocalTime` 语义要与标准保持一致。

## 审核清单
- 这次改动是否改错了 URI、HTTP 方法或 `IDList` 的使用方式。
- 这次改动是否只改了请求体，却没有同步检查 `ResponseStatus` 处理逻辑。
- 这次改动是否遗漏了 `User-Identify`、Digest 鉴权或平台要求的 HTTPS。
- 这次改动是否把元数据资源和 `Data` 资源混在一起处理。
- 这次改动是否只测了成功路径，没有验证 401、超时、重试、补传、重注册。
- 这次改动是否误把 `GA/T 1400.4` 当作字段定义来源，而忽略了 `GA/T 1400.3` / `GA/T 1399`。
