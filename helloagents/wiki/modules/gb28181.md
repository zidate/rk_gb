# GB28181

## 目的
沉淀 GB28181 在设备侧的注册、媒体、语音和控制实现方式。

## 模块概述
- **职责:** 负责 GB28181 注册、保活、实时预览、回放、下载、语音广播、语音监听、订阅通知和设备控制
- **状态:** ✅稳定
- **最后更新:** 2026-04-11
- **启动主链:** `CSofia(正常启动路径) -> ProtocolManager::Instance() -> ProtocolManager 私有 GB28181ClientSDK -> GB28181SDK -> SipSDK -> eXosip/oSIP`

## 产品需求基线

### 中国移动定制版终端要求
- `终端需求白皮书.pdf` 是当前工程的产品验收基线，语义上位于 `GB/T 28181-2016` 之上，不能被标准基线替代。
- 白皮书 `4.7.6.2` 明确“语音对讲”由“实时视音频点播 + 语音广播”两个独立流程组合实现，当前 App 层的广播桥/监听桥设计与这个模型一致。
- 白皮书附录 `A.1` 明确要求语音广播支持 `TCP 主动拉流`；白皮书附录 `A.14/A.15` 又将 `VoiceFlowMode` 定义为 `0=UDP，1=TCP主动拉流`。
- 白皮书附录 `A.16` 要求注册重定向时携带 `StringCode`、`Mac`、`CustomProtocolVersion` 等扩展信息，并在 `302` 后按指定节奏重试注册。

### 当前实现映射
- `GB28181BroadcastBridge::ApplySdpOffer/ApplyTransportHint/BuildLocalAnswerSdp` 已支持 `UDP / TCP active / TCP passive` 协商。
- `ProtocolManager::BuildGbBroadcastMediaInfo` 当前直接按 `m_cfg.gb_broadcast` 生成广播主动 `INVITE` 的媒体参数；`transport=tcp` 时固定回 `kRtpOverTcpActive`，不再复用旧广播会话里的协商缓存。
- 广播主动 `INVITE` 收到平台 `200 OK` 后，`ProtocolManager` 现会先把 answer 里的 `setup active/passive` 反解成本地运行角色，再把 transport 下发给 `GB28181BroadcastBridge`；平台回 `setup:passive` 时，设备会按 `tcp-active` 主动连接平台音频端口，而不是误切到本地 `listen`。
- `ProtocolExternalConfig::GbBroadcastParam.transport` 当前是白皮书 `VoiceFlowMode` 的最接近落点，但 `CfgBasicParam` / `DeviceConfigDownload` 并未把该字段作为查询/配置项显式暴露给平台。
- `ProtocolManager::StartGbClientLifecycle` 现在不会因为首次 `Register` 失败就直接退出；对于 SDK 返回的可恢复注册失败，设备会保留生命周期并交给后台线程按 `gb_keepalive.interval_sec` 继续重试。
- `ProtocolManager::StartGbClientLifecycle` 启动 SIP 客户端时已固定传本地端口 `0`，设备侧由内核分配随机本地 SIP 监听端口；`gb_register.server_port` 仅保留远端平台注册目标语义。
- `ProtocolManager::StartGbClientLifecycle` 当前已把 GB 客户端启动版本切到 `kGB2022Version`；`GBClientImpl -> SipClientImpl -> SipEventManager` 注册参数链路会在每次 `REGISTER` 上补标准头 `X-GB-Ver: 3.0`，同时保留零配置白皮书扩展头不变。
- `GB28181XmlParser` 对 `Keepalive.Info.DeviceID` 等原本仅在 `kGB2016Version` 下启用的扩展字段，现按“`2016` 及以上版本”复用，因此切到 `2022` 后不会把现有 fault-device 打包/解析能力带丢。
- `GB28181RtpPsSender` 的 TCP RTP/PS 发送路径现保留 `50ms` 单次 `SO_SNDTIMEO` 作为单次系统调用上限，但当 `send()` 返回 `EAGAIN/EWOULDBLOCK` 时不再立即失败，而是在 `3s` 总窗口内等待 socket 可写后继续发送；只有窗口耗尽或遇到连接重置等非重试型错误时，才向上返回失败并由 `ProtocolManager` 停流。
- `gb28181.ini` 新增 `register_mode=standard|zero_config` 运行时字段；`standard` 保持原有国标注册，`zero_config` 时 `ProtocolManager` 才会装配 `StringCode/redirect_server_id/redirect_domain` 等零配置参数。
- `ProtocolManager` 对启动/注册阶段的本地身份校验现按模式分流：`standard` 仍要求纯数字 GB 编码，`zero_config` 则允许白皮书场景下的字母数字 `StringCode`（例如 `C044...`）。
- `GBClientImpl::Register` 在 `use_zero_config=true` 时，已实现 `401 -> 302 -> 正式平台注册 -> 401 -> 200` 的单次零配置注册事务，并仅在 SDK 内存中缓存正式平台目标，供同一进程内的后续重注册复用，不回写 flash。
- `ProtocolManager::GbHeartbeatLoop` 仅在 `register_mode=zero_config` 时启用零配置外层重试节奏：正式平台注册失败 `30s` 重试，连续 `3` 次失败后 `1min` 并调用 `ResetZeroConfigState()` 重新获取重定向地址。
- `CGBClientImpl::Stop()` 会调用 `ResetZeroConfigState()` 清空零配置正式平台注册目标缓存；进程退出或设备重启后同样会失效，下次零配置启动仍基于 flash 中的入口参数重新走 `302`。
- `SipEventManager::GetClientInfo` 在处理 SIP 响应事件时，现优先用 `event->response` 中的平台侧 `To/Contact` 信息匹配 client，再回退到 `event->request`；这样可避免设备自发 `MESSAGE` 收到 `200 OK` 时，先把本机 `From` 误当 peer 而打印误导性的 `sip peer match failed`。
- `SipEventManager::StartSipListen` 在随机端口场景下会先临时 `bind(0)` 取回一个可用本地端口，再调用现有 `eXosip_listen_addr(actualLocalPort)` 建立正式监听，并把实际端口回写到 `ClientInfo.LocalPort`，确保 `REGISTER` 等出站请求的 `From/Contact` 携带真实监听端口。
- `ResponseGbQueryDeviceInfo` 已补齐白皮书 `A.19` 扩展身份字段，并回 `DeviceCapabilityList/ProtocolFunctionList` 最小真实能力集；其中 `FrameMirrorCapability`、`MultiStreamCapability`、`DeviceUpgradeCapability`、`AlarmCapability` 会直接暴露当前实现缺陷。
- `Catalog` 查询响应和目录订阅通知现共用 `PopulateGbCatalogInfo()` 生成目录项基础信息：`Name` 取运行态设备名，`Manufacturer/Model` 取 GB 配置，`CivilCode` 优先由国标编码前 `6` 位推导，`Address/IPAddress` 优先回设备对平台可达的本机 IP，`Owner` 当前以厂商名兜底，`RegisterWay` 固定回 `1`，`Parental/Secrecy/SafetyWay/Status/Event` 按当前单 IPC 最小实现回值。
- `GB28181XmlParser::PackCatalogResponse/PackCatalogNotify` 现会把 `CatalogInfo` 中的 `Owner/Ower`、`CivilCode`、`Address`、`Parental`、`RegisterWay`、`SafetyWay`、`Secrecy`、`IPAddress`、`Password`、`Event` 等字段真实编码进 XML，不再只回基础 `DeviceID/Name/Manufacturer/Model` 后让其他标签退成空串或默认值。
- `Catalog` 里的经纬度当前仍保持 SDK 默认 `0/0`：现有 `CatalogInfo` 结构没有 `Longitude/Latitude` 配置来源和字段承载，当前变更未扩展到该层，后续如平台强校验经纬度，需要先补充配置 schema 和 SDK 结构定义。
- `ProtocolManager::GetGbOnlineStatus()` 现对外暴露当前 GB28181 在线态，语义直接等同于 `m_gb_client_registered`，即设备是否已完成国标注册；启动中、重试中、已停止或配置关闭时都会返回 `false`。
- `App/Protocol/gb28181/GB28181ProtocolConstants.h` 已收口 GB28181 共用协议常量与映射：统一管理默认设备 ID、注册/保活缺省值、分辨率摘要、音频 payload/mime、`ConfigType/CmdType` 字符串以及 `FrameMirror/ImageFlip` 映射；`ProtocolManager`、`GB28181XmlParser`、`GB28181BroadcastBridge` 现复用同一套定义，避免协议名和值在多处漂移。
- `third_party/platform_sdk_port/CommonFile/CommonLib/ProtocolFeatureSwitch.h` 已删除；GB28181 Client SDK 现在作为当前产品线默认构建能力直接参与编译，`App/CMakeLists.txt`、`App/Protocol/gb28181/sdk_port/CMakeLists.txt` 中对应的重复 `-DPROTOCOL_HAS_GB28181_CLIENT_SDK=1` 也已去掉，`ProtocolManager` 历史上围绕这个宏的恒真条件编译分支只保留真实执行路径。
- `GB28181BroadcastBridge::DecodeToPcm16` 处理 `g711a/g711u` 时，现先把 `DG_decode_g711a/DG_decode_g711u` 的输出写入 `int16_t` 对齐临时缓冲，再拷回播放链使用的 PCM 字节流，避免直接把 `short*` 写进 `std::vector<char>` 带来的对齐风险。
- `GB28181BroadcastBridge` 现在按“每次会话独立”管理设备侧扬声器：旧连接关闭时会同步清掉 `AUDIO_TALK_TYPE` 播放态；新的 `UDP transport apply` 或 `TCP connect/accept` 成功后再调用一次 `StartAudioOut(AUDIO_TALK_TYPE)`。这样每轮广播/对讲在 SDP 协商和链路建立完成后都会重新确保扬声器打开，而不是靠每帧数据反复兜底。
- `ReconfigureGbLiveSender()` 与 `GB28181RtpPsSender` 现在显式区分 `tcp-active/tcp-passive`：当平台 `INVITE` 带 `setup:active` 时，设备侧会先监听本地 TCP 端口并在首包发送前按需 `accept`，`200 OK` 也会回填实际监听端口，不再把所有 TCP 预览都误当成设备主动 `connect`。
- `ProtocolManager::HandleGbLiveStreamRequest/ReconfigureGbLiveSender` 现在保持按 `a=streamnumber` / 本地默认流配置选择目标码流，再解析 `INVITE` SDP 里的 `f=` 视频参数并调用 `media::ApplyVideoEncodeStreamConfig()` 接口下发到该路码流；当前只先打通协议层入口和日志，编码器底层的完整动态重配仍待后续媒体侧补强。
- `ProtocolManager::HandleGbLiveStreamRequest / HandleGbLiveCaptureInternal / HandleGbStopStreamRequest` 现补充切流诊断日志：实时流请求会打印 `stream_num/stream/available_streams/default_stream`，会话替换会打印 `old_stream/new_stream`，采集错路丢帧按秒节流打印 `expected/incoming/drop_mismatch`，停流时汇总 `drop_mismatch` 计数，便于判断主子码流切换是否真的生效。
- `CAlarm::ThreadProc -> ProtocolManager::SyncLocalVideoAlarm -> NotifyGbAlarm` 已接通本地视频报警到 GB28181 平台的最小闭环；当前覆盖本地 `motion/human` 视频报警，并通过 `Info/AlarmTypeParam` 与 `Info/ExtraInfo` 承载细分描述。
- `HandleGbAlarmResetControl` 现在会同时清理本地报警录像触发标志和 `ProtocolManager` 内缓存的 GB 报警状态，但白皮书 `A.11` 中设备类扩展报警的“无需复位例外语义”仍未补齐。
- `GB28181XmlParser::UnPackGurdControl/UnPackRecordControl/UnPackAlarmResetControl/UnPackHomePositionControl` 现都会把请求 `<SN>` 回填给 SDK 统一业务应答链；`SetGuard/ResetGuard`、录像控制、告警复位、看守位控制的 `DeviceControl` 响应 XML 会保持与平台请求相同的 `<SN>`，避免回包落成脏值或旧流水号。
- SDK `OnAlarmSub` 的重复订阅删除已改为删除 `&m_alarm_sub.key`，避免旧报警订阅 timer/handle 残留。

## 规范

### 需求: 分析 GB28181 协议实现
**模块:** GB28181
需要梳理设备从协议启动到实时流、回放、广播和控制的完整闭环。

#### 场景: 平台拉实时流
前置条件:
- `ProtocolManager` 已启动
- `GB28181ClientSDK` 已绑定并完成注册
- 平台下发实时流请求
- 预期结果1: `GB28181ClientReceiverAdapter` 将请求转发给 `ProtocolManager`
- 预期结果2: `ProtocolManager` 先按 `a=streamnumber` 或本地默认流配置确定目标码流，再按 SDP `f=` 解析编码参数并下发到该路码流，然后启动对应采集回调
- 预期结果3: 音视频经 `GB28181RtpPsSender` 打包为 PS/RTP 发送

#### 场景: 平台请求语音广播
前置条件:
- 广播会话已启动
- 平台通过 SDP/传输参数协商音频端口与编码
- 预期结果1: `GB28181BroadcastBridge` 建立 UDP/TCP 接收通道
- 预期结果2: 接收 RTP 音频并解码为 PCM
- 预期结果3: 设备端播放广播音频或回调 PCM 数据

#### 场景: 设备语音监听回传
前置条件:
- 监听目标地址、端口、编码配置有效
- 预期结果1: 设备侧音频输入被启动
- 预期结果2: PCM 或 G711 音频按目标编码封装成 RTP
- 预期结果3: 发送到平台指定地址

#### 场景: 平台下发设备控制
前置条件:
- 设备已注册
- 平台下发控制消息
- 预期结果1: PTZ、图像翻转、OSD、升级、远程重启等命令被路由到设备能力
- 预期结果2: 需要持久化的配置通过 provider 回推

#### 场景: 设备本地报警上报
前置条件:
- `ProtocolManager` 已启动
- 平台已完成 `Alarm` 订阅
- 本地 `CAlarm` 检测到移动或人形告警
- 预期结果1: 设备保持本地录像/警笛等联动
- 预期结果2: `ProtocolManager::SyncLocalVideoAlarm` 只在报警状态变化时触发 `NotifyGbAlarm`
- 预期结果3: SDK 发送 `NOTIFY Alarm`，报文包含 `AlarmState`、`AlarmType`、`Info/AlarmTypeParam`、`Info/ExtraInfo`

## API接口

### `GB28181ClientReceiverAdapter::OnStreamRequest`
**描述:** SDK 流请求入口，负责分发实时流、回放、下载、停止流。
**输入:** `StreamHandle`、`gb_code`、`StreamRequestType`、`MediaInfo`
**输出:** 调用 `ProtocolManager::HandleGb*Request`

### `ProtocolManager::HandleGbLiveStreamRequest`
**描述:** 处理实时流请求并启动 DMC 采集回调。
**输入:** 流句柄、设备编码、媒体协商参数
**输出:** 建立实时流会话并响应媒体信息

### `ProtocolManager::HandleGbPlaybackRequest`
**描述:** 启动历史回放会话。

### `ProtocolManager::HandleGbDownloadRequest`
**描述:** 启动历史下载会话。

### `ProtocolManager::HandleGbPlayControl`
**描述:** 处理回放/下载 RTSP 控制，负责暂停、恢复、倍速、拖动和停止。

### `ProtocolManager::HandleGbDeviceControl`
**描述:** 处理设备控制命令，包括 PTZ、布防、告警复位、配置、升级、重启。

### OSD 对接说明
#### 现状结论
- 按 2026-03-30 当前代码，GB28181 的 OSD 获取和设置主链已通。
- 当前“GB 协议侧”和“设备 OSD 落地侧”之间已经抽出明确边界，实际链路为 `GB28181ClientReceiverAdapter -> ProtocolManager -> App/Media/VideoOsdControl -> Capture/rk_osd_*`。
- `ProtocolManager` 现在只负责 GB 协议字段映射、配置持久化和查询应答组包，不再直接管理 RK OSD 底层调用细节。
- OSD 的统一 get/set 结构现已收口到 `media::VideoOsdState`：`QueryVideoOsdState()` 返回它，`ApplyVideoOsdConfig()` 也直接接收它；文本项统一通过 `VideoOsdState.text_items[]` 承载。

#### 协议侧正式入口
- **查询入口:** `GB28181ClientReceiverAdapter::OnQuery -> ProtocolManager::HandleGbQueryRequest -> ProtocolManager::ResponseGbQueryConfig`
- **设置入口:** `GB28181ClientReceiverAdapter::OnDeviceControl -> ProtocolManager::HandleGbDeviceControl -> ProtocolManager::HandleGbConfigControl`
- **白皮书兼容入口:** 设备侧已补齐 `ConfigDownload + ConfigType=OSDConfig` 查询识别、`ConfigDownload` 应答中的 `<OSDConfig>` 编码，以及 `DeviceConfig + OSDConfig` 的设置解析。
- **协议字段范围:** 当前设备同时兼容两条 OSD 配置链路：
  - 标准 SDK 旧模型：`SVACEncodeConfig/SVACDecodeConfig` 下的 `TimeShowFlag`、`EventShowFlag`、`AlertShowFlag`
  - 白皮书模型：`OSDConfig.Length/Width/TimeX/TimeY/TimeEnable/TimeType/TextEnable/SumNum/Item(Text,X,Y)`

#### 当前设备侧实际落地路径
- `ProtocolManager::HandleGbConfigControl` 收到 `SVACEncodeConfig/SVACDecodeConfig` 或白皮书 `OSDConfig` 后，会先通过 `media::QueryVideoOsdState()` 获取当前 OSD 逻辑状态，在协议层 merge 本次 delta，再按最终目标配置决定 `skip_apply` 还是调用 `media::ApplyVideoOsdConfig()` 下发到媒体/编码侧；GB 模块不再回写 `m_cfg.gb_osd.*` 作为运行态缓存。
- 白皮书 `OSDConfig` 的 `TextEnable/SumNum/Item[]` 现会直接映射到 `VideoOsdState.has_text_enabled/text_enabled/text_items[]`；协议层比较与下发都围绕统一的 `VideoOsdState` 执行，不再维护单独的 `VideoOsdConfig` 中间模型。
- `ProtocolManager::ResponseGbQueryConfig` 收到 `ConfigType=OSDConfig` 时，会通过 `media::QueryVideoOsdState()` 读取设备 OSD 运行态（总开关、时间开关、`event/alert` 逻辑标志、文本开关、时间样式、时间位置、多文本项），再回组装白皮书 `<OSDConfig>` 的 `SumNum + Item[]`，不再以 `m_cfg.gb_osd` 作为正常回包来源。
- `App/Media/VideoOsdControl.*` 是当前设备侧 OSD 统一接口面，内部默认适配的设备能力包括：
  - `CaptureSetOSDSwitch`
  - `rk_osd_set_enabled`
  - `rk_osd_set_display_text`
  - `rk_osd_set_date_style`
  - `rk_osd_set_time_style`
  - `rk_osd_set_position_x / rk_osd_set_position_y`
  - `rk_osd_restart`
- 对于底层设备没有独立 getter 的 `event/alert` 逻辑位，以及必要的 OSD 总开关兜底状态，现统一下沉到 `VideoOsdControl` 内部维护；缓存归媒体模块自己负责，不再放在 GB 模块。
- 因此，对接方如果只关心“GB 平台设置能否在设备上生效”，当前答案是可以；并且设备侧落地接口已经从协议模块中抽离出来，后续编码模块维护者可以直接在 `VideoOsdControl` 内替换实现。
- 当前 OSD 控制侧和画面反转一样，GB 模块已经显式承担 `query -> compare -> apply` 边界；即使媒体内部仍保留兜底判断，协议层日志中也能直接看到运行态和目标值的比对结果。

#### 当前可被其他模块使用的配置模型
- `ProtocolExternalConfig::gb_osd` 仍保留为外部配置 schema，字段包括：
  - `text_template`
  - `time_format`
  - `position`
  - `time_enabled`
  - `event_enabled`
  - `alert_enabled`
- 这套 `gb_osd` 结构目前仍是历史兼容配置模型，只用于外部配置输入；它不是当前 OSD 运行态的统一 get/set 结构，也不承载多文本 `Item[]`。
- HTTP 配置侧已经支持上述字段的读写，字段名分别为：
  - `gb_osd_text_template`
  - `gb_osd_time_format`
  - `gb_osd_position`
  - `gb_osd_time_enabled`
  - `gb_osd_event_enabled`
  - `gb_osd_alert_enabled`
- 但按 2026-03-28 当前代码，GB 模块不再把 `gb_osd` 作为运行态缓存或查询应答来源；`ProtocolManager::GetConfig().gb_osd` 不能再视为“设备当前 OSD 真实状态”。
- 当前 `ProtocolManager` 对外公开的稳定入口仍以 `GetConfig()`、`ReloadExternalConfig()`、注册配置读写/重启接口为主，还没有单独的 `GetGbOsdConfig`、`SetGbOsdConfig`、`ApplyGbOsdConfig` 公共方法。

#### 其他模块对接建议
- 如果其他模块只是**读取当前设备 OSD 运行态**，应直接通过 `App/Media/VideoOsdControl.h` 的 `QueryVideoOsdState()` 获取，不要再把 `ProtocolManager::GetConfig().gb_osd` 当作运行态来源。
- 如果其他模块只是**修改配置源**，建议走现有外部配置通道，由 `ProtocolManager::ReloadExternalConfig()` 或现有配置变更流程统一生效。
- 如果编码/媒体模块要**直接承接设备 OSD 实现**，当前应以 `App/Media/VideoOsdControl.h` 暴露的接口为准，在媒体模块内部实现或替换底层能力。
- 如果其他模块要**直接驱动设备 OSD 生效**，当前不要绕过 `ProtocolManager` 自己拼 GB 协议处理逻辑，也不要把 `DevInterface` 的 OSD 四个虚接口直接视为当前 GB OSD 的正式接入点。

#### 当前未作为 GB OSD 正式入口的接口
- `DevInterface` 虽然定义了：
  - `GetOsdTimestamp`
  - `GetOsdChnName`
  - `SetOsdTimestamp`
  - `SetOsdChnName`
- 但截至 2026-03-28 当前仓库实现，GB28181 OSD 主链没有调用这些接口。
- 对接上应把它们视为“底层设备抽象预留能力”，而不是“当前 GB OSD 已接入并联调通过的标准接口”。

#### 当前边界
- 白皮书 `OSDConfig` 现已打通设备侧查询/设置闭环，协议态 `VideoOsdState` 也已经支持多文本 `text_items[]`。
- 底层 RK OSD 真实能力仍只有“一套公共位置 + 一条自定义文本”；因此媒体层当前只会尽力把第 `1` 个文本项及其坐标落到底层设备。
- 当平台设置多个 `Item` 时，额外文本项会保存在 `VideoOsdState` 的协议态缓存中，并在后续 `ConfigDownload + OSDConfig` 查询时继续通过 `SumNum + Item[]` 回显；这属于协议兼容保留，不代表 SoC 已原生支持多文本同时显示。
- `TimeType=1` 会在协议配置和查询回显中保留，但 RK OSD 底层不原生支持中文日期样式，实际显示仍会回落到当前可用的日期/时间样式。
- `kEncodeSetting` 与 `kDecodeSetting` 当前最终都落到同一套全局 `VideoOsdState` / 设备 OSD 状态，没有做独立通道区分。

### 编码参数对接说明
#### 现状结论
- 按 2026-03-28 当前代码，GB28181 侧读取编码参数的正式路径已经切到 `GB28181ClientReceiverAdapter -> ProtocolManager -> App/Media/VideoEncodeControl -> rk_video_*/CaptureGetResolution`。
- `ProtocolManager` 不再直接访问 `rk_video_get_output_data_type`、`rk_video_get_frame_rate`、`rk_video_get_max_rate`、`rk_video_get_gop`、`CaptureGetResolution` 这些底层编码接口。
- 设备侧统一接口面已经固定为 `App/Media/VideoEncodeControl.*`，当前暴露：
  - `QueryVideoEncodeStreamState`
  - `QueryVideoEncodeState`
  - `ApplyVideoEncodeStreamConfig`
- 编码参数 XML 已按 `GB/T 28181-2022` 对齐到 `VideoParamAttribute`，不再把 `codec/fps/bitrate` 继续塞在旧的 `VideoParamOpt/SVACEncodeConfig` 语义里。

#### 当前协议侧使用方式
- `ProtocolManager::ResponseGbQueryConfig` 现在通过 `media::QueryVideoEncodeState()` 读取主辅码流运行态，并按 `VideoParamAttribute` 回包：
  - `StreamNumber`
  - `VideoFormat`
  - `Resolution`
  - `FrameRate`
  - `BitRateType`
  - `VideoBitRate`
- `ProtocolManager::HandleGbConfigControl` 现在支持解析 `DeviceConfig + VideoParamAttribute`，并会先显式调用 `media::QueryVideoEncodeState()` 读取主辅码流运行态，再按流决定 `skip_apply` 还是通过 `media::ApplyVideoEncodeStreamConfig()` 下发到编码侧；GB 模块不再把 `m_cfg.gb_video` 当作编码参数运行态缓存。
- `GB28181XmlParser` 已补齐：
  - `ConfigDownload + ConfigType=VideoParamAttribute` 查询识别
  - `ConfigDownload` 应答里的 `<VideoParamAttribute>` 编码
  - `DeviceConfig` 控制里的 `<VideoParamAttribute>` 解析
- `ProtocolManager::BuildGbResponseMediaInfo` 在实时流协商时通过 `media::QueryVideoEncodeStreamState()` 读取实际编码类型，用于确定当前会话的 `video_codec`。
- `BuildGbStreamNumberList` 与 OSD 画布尺寸推导也不再直接碰编码底层分辨率接口，统一通过媒体侧查询主/辅码流分辨率。

#### 当前边界
- 当前 `VideoParamAttribute` 的标准映射采用 `GB/T 28181-2022` 附录 G 口径：
  - `VideoFormat`: `1=MPEG4`、`2=H.264`、`3=AVS`、`4=SVAC`、`5=H.265`
  - `Resolution`: `1=QCIF`、`2=CIF`、`3=4CIF`、`4=D1`、`5=720P`、`6=1080P/I`，其他分辨率回 `WxH`
  - `BitRateType`: `1=固定码率(CBR)`、`2=可变码率(VBR)`
- `VideoEncodeControl` 现已把码率控制模式一起纳入媒体接口查询/设置；GB 模块不直接访问 `rk_video_get_RC_mode/rk_video_set_RC_mode`。
- 当底层编码 getter 暂时不可用时，最小兜底状态也应由 `VideoEncodeControl` 自己维护；GB 查询/控制侧只认媒体接口，不再回退到 `m_cfg.gb_video`。
- 受当前设备能力限制，`StreamNumber` 仍只支持 `0=主码流`、`1=辅码流` 两路；收到其他码流序号会记录日志并忽略。
- 如果后续编码模块由其他同事维护，应直接在 `VideoEncodeControl` 内部替换 RK 默认实现，而不是回到 `ProtocolManager` 中重新拼 `rk_video_*` 调用。
- 当前编码参数控制侧与画面反转、OSD 一致，GB 模块已经显式把媒体查询和是否真正下发设置的决策暴露出来，便于代码审查和联调定位。

### 画面反转对接说明
#### 现状结论
- 按 2026-03-28 当前平台联调抓包，画面反转正式口径使用的是顶层 `DeviceConfig + FrameMirror` / `ConfigDownload + FrameMirror`，不是旧的 `VideoParamOpt/ImageFlip`。
- `FrameMirror` 当前按平台抓包顺序对齐为：`0=还原(close)`、`1=左右(mirror)`、`2=上下(flip)`、`3=中心(centrosymmetric)`。
- GB28181 侧读取 / 设置画面反转的正式路径已经切到 `GB28181ClientReceiverAdapter -> ProtocolManager -> App/Media/VideoImageControl -> rk_isp_get_image_flip/rk_isp_set_image_flip`。
- `ProtocolManager` 不再直接读写 `CFG_CAMERA_PARAM.rotateAttr` 或调用底层翻转接口；协议侧只负责 `FrameMirror` 数值和设备侧字符串模式之间的映射。
- 为兼容旧链路，`VideoParamOpt/ImageFlip` 仍保留解析 / 查询回包，但平台联调应以 `FrameMirror` 为准。

#### 当前协议侧使用方式
- `GB28181XmlParser` 已补齐：
  - `ConfigDownload + ConfigType=FrameMirror` 查询识别
  - `ConfigDownload` 应答里的 `<FrameMirror>` 编码
  - `DeviceConfig` 控制里的 `<FrameMirror>` 解析
- `ProtocolManager::ResponseGbQueryConfig` 现在支持按运行态回 `FrameMirror`，同时保留旧 `VideoParamOpt.ImageFlip` 查询兼容；GB 模块不再回退到 `m_cfg.gb_image.flip_mode`。
- `ProtocolManager::HandleGbConfigControl` 收到 `FrameMirror` 后，会先显式调用 `media::QueryVideoImageFlipMode()` 读取当前运行态，再统一转成 `close/mirror/flip/centrosymmetric`，按需通过 `media::ApplyVideoImageFlipMode()` 下发到媒体侧。

#### 当前边界
- 设备侧统一接口面已经固定为 `App/Media/VideoImageControl.*`，当前暴露：
  - `QueryVideoImageFlipMode`
  - `ApplyVideoImageFlipMode`
- 图像翻转的必要兜底状态现由 `VideoImageControl` 自己维护，避免 GB 模块再长期保存一份独立镜像值。
- 如果后续画面反转由编码 / ISP 模块同事接管，应直接在 `VideoImageControl` 内部替换 RK 默认实现，而不是回到 `ProtocolManager` 中恢复底层直调。

### `ProtocolManager::SyncLocalVideoAlarm`
**描述:** 将 App 本地视频报警状态变化同步到 GB28181 报警通知链路，并做去重控制。

**输出:** 有报警订阅时触发 `NotifyGbAlarm`；无订阅时快速返回

## 实现要点
- **注册与保活:** `ProtocolManager::StartGbClientLifecycle -> RegisterGbClient -> GbHeartbeatLoop`
- **冷启动恢复:** 首次注册返回可恢复错误时，`StartGbClientLifecycle` 只记录 `note=defer_retry`；`GbHeartbeatLoop` 在未注册态下会按 `gb_keepalive.interval_sec` 继续尝试注册
- **实时流:** `HandleGbLiveStreamRequest -> StartGbLiveCapture -> PushLiveVideoEsFrame/PushLiveAudioEsFrame -> live GbMediaSenderRuntime`
- **回放与下载:** `HandleGbPlaybackRequest/HandleGbDownloadRequest -> StartGbReplaySession -> Storage_Module_* -> replay GbMediaSenderRuntime`
- **回放控制:** `HandleGbPlayControl` 现已将 `PAUSE` 映射为暂停、`PLAY` 映射为恢复/倍速/拖动并恢复发流、`TEARDOWN` 映射为停止回放会话
- **时间基线:** `StartGbReplaySession` 直接使用 SDP/SDK 传入的 `StartTime/EndTime`，不再对回放时间硬编码追加 `+28800`
- **媒体并发模型:** 自 2026-03-27 起，GB 实时预览与 GB 回放/下载使用两套独立 sender 和独立 SSRC/local port，支持同时并发；当前仍只支持 `1` 路 live 与 `1` 路 replay/download，并保持“新同类请求替换旧同类会话”的语义
- **停流精度:** `HandleGbStopStreamRequest` 现按 `handle` 精确命中 live/replay/talk/broadcast，会话 stop 不再因另一类媒体句柄为空而被误停
- **广播下行:** `HandleGbAudioStreamRequest -> GB28181BroadcastBridge`，支持 SDP/传输协商、UDP、TCP active/passive
- **监听上行:** `GB28181ListenBridge` 使用本地音频输入与 DMC 音频回调，将音频编码后 RTP 发往目标地址
- **本地报警上报:** `CAlarm::ThreadProc -> ProtocolManager::SyncLocalVideoAlarm -> NotifyGbAlarm -> NotifyDeviceAlarmForSub`，当前按 `VideoMotion` 标准类型上报，并用 `AlarmTypeParam=motion/human/vehicle` 承载本地细分语义
- **基础参数查询/配置:** `ResponseGbQueryConfig` / `HandleGbConfigControl` 当前主要支持 `DeviceName`、`Expiration`、`HeartBeatInterval`、`HeartBeatCount`、`FrameMirror/ImageFlip(兼容)`、OSD 相关字段
- **设备信息查询:** `ResponseGbQueryDeviceInfo` 当前返回 `device_name/type/manufacturer/model/firmware/video_channel_num`，并追加 `StringCode/Mac/Line/CustomProtocolVersion`、`DeviceCapabilityList`、`ProtocolFunctionList`

## 注意事项
- App 层主要负责设备能力适配和媒体桥接，真正的 SIP/GB28181 协议处理主体在移植后的 `GB28181SDK + SipSDK + eXosip/oSIP`。
- 当前代码清晰实现了“平台广播下行”和“设备监听上行”两条音频链路。
- `ProtocolExternalConfig` 中定义了 `gb_talk`，但在当前 App 层代码中未形成完整独立的双向对讲会话控制闭环，后续联调时不要把“广播/监听”直接等同于“标准对讲”。
- 白皮书要求广播支持 `TCP 主动拉流`，公网平台联调时要优先核对平台 `INVITE/应答` 中返回的媒体 `IP/PORT` 和 `setup` 方向。
- 当前工程已按代码路径落地白皮书 `30s / 3次 / 1min / 重新获取重定向地址` 的重定向状态机，但还缺真实平台抓包验证，不宜在联调前直接视为最终验收通过。
- `gb28181.ini` 中 `register_mode` 默认值为 `standard`；只有切到 `zero_config` 时才会发送零配置扩展头并进入 `302` 重定向注册流程。
- 零配置运行期通过 `302` 获取的正式平台 `ServerIp/ServerPort/ServerDomain/ServerId/deviceId` 当前只保存在 SDK 内存，不会持久化到 `gb28181.ini` 或 `zero_config.ini`。
- 白皮书 `A.11` 内部对扩展设备报警编码存在 `900-904` 与 `6-10` 两套写法，必须先和平台确认最终口径，再决定 `AlarmType` 取值。
- 当前 `AlarmNotifyInfo` 已补齐 `AlarmTypeParam`，XML `Alarm` 也已支持 `Info/ExtraInfo` 最小承载；但 `AlarmTypeDevice` 枚举仍只覆盖 `1-5` 基础设备报警码，若要做白皮书扩展设备告警，仍需先确定最终编码口径。
- `NotifyDeviceAlarmForSub` 当前使用 `Notify(..., 0, NULL)`，没有业务等待，更接近白皮书“无应答响应方式”口径，但仍应结合平台联调确认是否只需要 SIP 标准应答。
- 当前已补齐“本地视频报警 -> GB28181 平台报警”的最小闭环，但“掉电/升级失败/文件上传失败”等设备类扩展告警事件源仍未接入。
- 白皮书 `A.14/A.15` 的 `VoiceFlowMode`、`LocalPort`、`Domain`、`SipIp`、`SipPort`、`SipId`、`Password`、`ChannelList` 当前没有被 `CfgBasicParam` / `BasicSetting` 完整承载。
- 白皮书 `A.19` 的扩展身份字段和最小能力节点已落入 `DeviceInfo`，但当前仍只覆盖最小真实子集；若平台要求更完整的附录 `G` 子项，仍需继续扩展。
- 白皮书 `4.7.9` 要求三码流或以上，而当前 `GbMultiStreamParam.stream_count` 默认值为 `2`，后续若以白皮书为验收口径，需要单独补齐。
- `GB28181RtpPsSender` 在编译期依赖 `third_party/media-server/libmpeg/include` 和 `third_party/media-server/librtp/include` 下的头文件；运行期仍通过 `dlopen("libmpeg.so")` 和 `dlopen("librtp.so")` 动态装载实际库文件。
- 当前仓库已补齐 `mpeg-ps.h`、`mpeg-proto.h`、`rtp-payload.h`、`rtp-profile.h` 这组编译期公开头，用于保证 GB28181 PS/RTP 发送链路可以完成交叉编译。
